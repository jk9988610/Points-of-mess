# 论证引擎路线图（简便 · 自然对话 · 可伸缩 N 推 M）

> **产品核心**：在对话里**一条一条拿到证据**，**一个一个闭合论断**，直到**本局论证目标**成立。  
> **「三推一」不是固定版式**，而是**逻辑骨架的俗称**——同一引擎可配置为 **2 推 1**、**3 推 1**、**3 推 2** 等。

---

## 1. 概念重命名（对用户与 prompt 统一口径）

| 旧说法（易误解为固定条数） | 新说法（逻辑角色） |
|---------------------------|-------------------|
| 3 条 `[已确认]` 种子 | **前提 / 已采纳证据**（`premises[]`，开局可 2～4 条） |
| 1 条 `[待核实#1]` | **开放论断**（`openClaims[]`，可同时 1～2 条，上限由 preset 定） |
| 【本局目标】 | **论证目标**（`argumentGoal`，唯一终局问句） |
| keypoint 亮牌交换 | **举证回合**（玩家出价 → 锋利须兑现一条可核对证据） |
| 摘要 `[已确认]` 增多 | **证据入库**（前提变厚，开放论断被回答则闭合或改写） |

**一局 = 一个目标 + 若干前提 + 若干待闭合论断**；条数是 **preset 参数**，不是全局常量。

---

## 2. 逻辑骨架示例（preset 级）

### 2.1 锋利 · 标准局（现网近似 **3 推 1**）

```text
论证目标：谁在背后操控（指使 + 账本链）
前提（种子）：阻拦事实、陈四、刘老三经手 …
开放论断（1）：陈四的指使者是谁
闭合路径：举证回合 ×2（陈四轨 → 刘老三轨）→ 主使槽闭合 → 结局
```

### 2.2 短局（**2 推 1**）

```text
前提（2）：锋利被拦；账本在刘老三
开放论断（1）：指使者是谁
minKeypointTurns: 1
```

### 2.3 双线局（**3 推 2**）

```text
前提（3）：…
开放论断（2）：
  #A 指使者是谁
  #B 账本现下落
结局：A、B 均闭合（或一条被程序判定为「已由前提蕴含」而自动闭合）
```

**规则**：开放论断 **≤ `maxOpenClaims`**（建议 1～2，硬顶 3 仅给实验 preset）；前提 **≤ `maxPremises` 展示**（超出折叠进【前提·背景】）。

---

## 3. 运行时不变量（比「几条」更重要）

1. **一轮一证**：keypoint 回合锋利只兑现 **一条** 新证据（已有，保持）。
2. **证据可核对**：入库行须含专名 / 可验证陈述，禁止把「态度」当证据。
3. **槽位单真值**：同一 slot（如 `mastermind`）改口 → **覆盖或 [已推翻]**，禁止双主使并存。
4. **结局 = 论证闭合**：  
   `argumentGoal` 可推导 + **无未闭合的开放论断**（或 preset 声明某条可残留）+ **无未解决槽位矛盾** + **最少举证回合数**。
5. **自然感**：followup 允许短拒、顶回；**不强制每轮送料**；推进靠玩家选 keypoint，不靠摘要灌水。

## 4. 配置模型（`onionSeed` 演进）

```javascript
argumentProfile: {
  /** 展示用标签，如 "3推1" / "2推1" / "3推2" */
  label: "3推1",
  maxPremises: 8,        // 档案中 [已确认] 上限（含局内新增）
  maxOpenClaims: 1,      // 并行 [待核实] 上限
  minPremisesForEnding: 3,
  minKeypointTurns: 2,
},
argumentGoal: "查明：…",  // 原 goal
premises: [ ... ],         // 原 confirmed（种子）
openClaims: [ ... ],       // 原 pending（可多条）
evidenceSlots: [           // 程序判定用，可选
  { id: "mastermind", closes: ["指使者", "主使"] },
  { id: "ledger", closes: ["账本", "下落"] },
],
```

**兼容**：未写 `argumentProfile` 时，用 `confirmed` / `pending` / `endingMinConfirmed` 推断（现 v0.5.41 行为）。

## 5. 优先级实施计划（按简便性排序）

### 阶段 A — 论证闭合引擎（**最高优先级**）

| 项 | 内容 | 为何先做 |
|----|------|----------|
| A1 | `maxOpenClaims` 从 seed 读取；`extractPendingLines` / 摘要校验改为 **≤N** 而非写死 1 | 解锁 2 推 1 / 3 推 2，改动面小 |
| A2 | `reconcileEvidenceSlots`：主使/账本改口覆盖；删 `待核实 … 无` | 直接修 v0.5.41 坏档，结局才可信 |
| A3 | `isArgumentClosed(plot, seed)` 替代「数 [已确认] 条数」：看 **开放论断是否空** + **必填 slot 是否闭合** | 结局与「论证完成」一致 |
| A4 | followup **短拒白名单**（如「谁都不保」且非 keypoint） | 自然对话，少误拒 |

**验收**：`verify-archive-contradiction.js`；人工一局改口后档案单主使；2 推 1 preset 可 1 次 keypoint 后仍不能结局（若 `minKeypointTurns:2`）。

**版本建议**：v0.5.42（A1–A4 可一个 PR）。

---

### 阶段 B — 预设与 prompt 去「固定三推一」

| 项 | 内容 |
|----|------|
| B1 | `summary.js` / `DIALOGUE-PROMPT.md`：改为「论证目标 + 前提 + 开放论断（至多 N）」 |
| B2 | 新增 `sharp-short` preset：**2 推 1**（`openClaims.length=1`, `minKeypointTurns=1`） |
| B3 | 可选 **3 推 2**：`openClaims: [指使者, 账本下落]`，`maxOpenClaims:2` |

**验收**：`verify-onion-seed.js` 接受 2 条 pending；摘要 API 不再 warn「应为至多 1 条」当 N=2。

---

### 阶段 C — 结局变体（重玩价值，依赖 A2/A3）

| 项 | 内容 |
|----|------|
| C1 | `endingVariant`：`success` / `fail_neglect` / `variant_cover`（半真相） |
| C2 | epilogue 模板按 variant 注入结局 API |

**判定要点**：未质疑主使矛盾就收束 → `variant_cover`；论证闭合且无推翻残留 → `success`。

---

### 阶段 D — 交叉论证（**3 推 2+** 的深化，最后做）

| 项 | 内容 |
|----|------|
| D1 | `claimDependencies`：论断 B 依赖 A |
| D2 | 档案双名冲突 → 选项注入「认哪一个」 |
| D3 | `crossVerified` 才进 `success` |

**前置**：A2 槽位稳定，否则并行论断与矛盾档案叠加更乱。

## 7. 明确不做（保持简便）

- 不把「必须恰好 3 条种子已确认」写进程序硬门槛（改为 `minPremisesForEnding` + slot）。
- 不为每局动态生成 5+ 开放论断（模型会爆摘要）。
- 第一阶段不合并 ①②③ API。
