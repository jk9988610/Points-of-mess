## 2. 目标输出格式（定稿）

模型只输出摘要正文，**严格**两段标题：

```text
【剧情档案】
- [已确认] …
- [待核实] …
- [已推翻] …（极少；与 [已确认] 冲突时注明旧说法待废弃，新事实仍用 [已确认]）

【关系与态度】
- …
```

### 2.1 行级标记语义

| 标记 | 含义 | 规则 |
|------|------|------|
| `[已确认]` | 对话中已说定、可核对的事实 | 只增不删专名行；冲突时保留旧行并加 `[待核实]` 新问句 |
| `[待核实]` | 仍未得到明确答案的问题 | **必须**具体、可回答；禁止「是否会…」类元问题 |
| `[已推翻]` | 明确被否定/证伪的旧说法 | 可选；不替代 [已确认] 的专名删除 |

### 2.2 压缩规则（写入 `SUMMARY_SYSTEM`）

1. **递进合并**：输入「已有摘要 + 新增对话」→ 在旧档案上更新，**禁止**整体重写变短。  
2. **待办维护**：本轮已明确回答的 `[待核实]` → **删除**或**改写**为更细的 `[待核实]`。  
3. **新事实**：写入 `[已确认]`，无需另开「本轮新增」段。  
4. **篇幅**：全文 ≤ `SUMMARY_MAX_CHARS`（1200）；优先压缩/合并 `[待核实]` 条数，**不**删 `[已确认]` 专名。  
5. **关系段**：无新变化可写「暂无变化」。  
6. **禁止**：markdown 代码块；无标题的散文块。

## 5. `SUMMARY_SYSTEM` 草稿（实现时粘贴微调）

```text
你是剧情摘要压缩助手。根据「已有摘要」与「新增对话」，输出更新后的结构化摘要。

【输出格式】严格两段：
【剧情档案】
- [已确认] …
- [待核实] …
（可选）[已推翻] …

【关系与态度】
- …

【规则】
1. 在已有摘要基础上递进更新；禁止整体变短；禁止删除已有 [已确认] 行中的专名（人名、地点、物证）。
2. [待核实] 必须具体、可回答；禁止「是否会…」类元问题。
3. 新增对话中已明确回答的 [待核实]：从待办中删除，或将要点写入 [已确认]；部分回答则改写为更细的 [待核实]。
4. 新事实用 [已确认] 追加；不要单独写「本轮新增」段。
5. 全文不超过 {SUMMARY_MAX_CHARS} 字；优先合并待核实条数，不删已确认专名。
6. 只输出摘要正文，不要 markdown 代码块或其它说明。
```

## 6. 示例（档案一行一事）

**锋利上一窗口后（摘录）**

```text
【剧情档案】
- [已确认] 监控显示林晨凌晨三点离开厂区东门
- [已确认] 玩家持有蓝色账本封面照片，锋利否认见过实物
- [待核实] 蓝色账本完整下落与最后经手人
- [待核实] 林晨离开东门后去向

【关系与态度】
- 锋利对玩家施压，回避直接点名内鬼
```

**玩家本轮问清「账本在城西废弃仓」后，下次压缩应变为**

```text
- [已确认] 蓝色账本现藏于城西废弃仓（锋利本窗口承认）
- [待核实] 谁把账本放到废弃仓、是否与林晨有关
```

（「账本在哪」不再单独占一条 `[待核实]`。）

## 8. 风险与回滚

| 风险 | 缓解 |
|------|------|
| 模型不按 `[待核实]` 输出 | prompt 示例 + 压缩后本地检测；无标记时打 `logLocalWarn`，摘录走旧段回退 |

| 旧存档四段格式 | `extractPendingVerification` 双格式回退（V6） |

回滚：恢复旧 `SUMMARY_SYSTEM` + 旧 `extractUnresolved` 即可（单 PR revert）。
